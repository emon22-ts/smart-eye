System Overview

Multi-Modal Ocular Health Screening

Smart Eye fuses three independent data sources — a CNN image classifier, a live webcam fatigue monitor, and a symptom questionnaire — through a Mamdani fuzzy inference engine to produce a single explainable Ocular Health Index (OHI).

Input 1
🖼 Fundus Image
User uploads an anterior-segment or fundus photograph. EfficientNetB0 classifies into 4 disease categories and produces a Grad-CAM heatmap.
EfficientNetB0 Grad-CAM 4-class CNN
Input 2
👁 Webcam Fatigue
Live 68-point facial landmark tracking computes Eye Aspect Ratio (EAR) at ~8 Hz to detect drowsiness and measure blink compliance in real time.
dlib 68-pt EAR tracking ~8 Hz
Input 3
📋 Symptom Survey
A structured Likert-scale questionnaire captures subjective symptom burden (blurred vision, pain, photophobia, etc.) and produces a 1–5 aggregate severity score.
Likert 1–5 7 symptoms
↓   ↓   ↓
Mamdani Fuzzy Inference Engine
Decoupled PRI + SRI Fusion
Two isolated evaluation tracks ensure that a high-confidence pathology finding always triggers immediate triage, regardless of fatigue level. Centroid defuzzification → OHI [0–100].
PRI — Pathology Risk Index SRI — Screening Reliability Index Centroid defuzz scikit-fuzzy
↓
OHI
Ocular Health Index
Composite 0–100 score. Higher = healthier. Displayed as an animated gauge with Low / Moderate / High risk bands.
Triage
Clinical Recommendation
One of: ROUTINE_MONITORING · FOLLOW_UP_RECOMMENDED · IMMEDIATE_CLINICAL_PRIORITY. Derived from PRI only — fatigue-independent.
CAM
Grad-CAM Heatmap
Gradient-weighted class activation map overlaid on the uploaded image, highlighting the regions driving the CNN prediction.
Layer Architecture

Five-Layer Stack

Smart Eye follows a clean separation of concerns across five distinct layers from user interface to data persistence.

Layer 1 — Presentation
User Interface (React 18 + Vite)
Single-page application served on port 5173. Handles all rendering, routing, theme management, i18n (EN/বাংলা), and accessibility. Communicates with the backend exclusively via REST API calls — no direct database or model access.
React 18 Vite Vanilla CSS React Router 8 pages Dark + Light theme EN / বাংলা i18n
↕
Layer 2 — API Gateway
FastAPI Application (smart_eye/orchestration/api.py)
Async REST API on port 8000. Handles authentication (JWT + Google OAuth), request validation (Pydantic), session orchestration, and routing to domain services. Mounts a parallel `/api/anterior/*` router for the anterior-segment module.
FastAPI Pydantic JWT auth Google OAuth Python 3.9.6 Uvicorn
↕
Layer 3 — Orchestration
Session Orchestrator (score_session)
Coordinates the three-stage screening workflow: (1) invoke CNN inference on the uploaded image, (2) collect fatigue snapshot, (3) receive symptom score, then feed all three into the fuzzy engine and return a unified OHI result with triage recommendation.
Multi-modal fusion FatigueSnapshot OHI computation Recommendation engine
↕
Layer 4 — Domain Services
Core Intelligence (smart_eye/domain/)
CNN Pipeline
EfficientNetB0 inference · 4-class softmax · Grad-CAM heatmap · pathology confidence
Fatigue Monitor
68-point landmark EAR · blink detection · drowsy flag · per-session snapshot
Fuzzy Engine
Mamdani PRI + SRI · Centroid defuzz · NaN guard · decoupled clinical safety
TensorFlow 2.13 scikit-fuzzy dlib NumPy fpdf2 OpenCV

Layer 5 — Persistence
Local SQLite Storage (smart_eye/persistence/store.py)
Privacy-first on-device storage. Only numerical scores and timestamps are persisted — no images, no raw biometrics. Two tables: users (hashed credentials) and sessions (OHI, risk band, top finding, fatigue score, symptom aggregate). No cloud transmission.
SQLite bcrypt hashing No image storage GDPR-compliant Local only

Fuzzy Inference Engine

Decoupled Mamdani System

The clinical safety requirement mandates that high pathology risk always triggers immediate triage, regardless of the user's fatigue state. This is achieved through structural decoupling — two isolated Mamdani systems with no shared outputs.

cnn_confidence
Range: [0.0 – 1.0] · 1 − P(Normal)
symptom_burden
Range: [1.0 – 5.0] · Likert aggregate
→
PRI Engine
Pathology Risk Index
MFs: Triangular + trapezoidal
Rules: 3×3 expert block (9 rules)
Defuzz: Centroid
Shoulders: edge-pinned (no NaN)
→
PRI [0–100]
→ Triage recommendation
PRI ≥ 66 → IMMEDIATE
PRI ≥ 34 → FOLLOW_UP
PRI < 34 → ROUTINE
eye_fatigue
Range: [0 – 100] · EAR-derived score
blink_compliance
Range: [0 – 100] · blink rate normalised
→
SRI Engine
Screening Reliability Index
MFs: Triangular + trapezoidal
Rules: 3×3 expert block (9 rules)
Defuzz: Centroid
NaN guard: fallback = 50
→
SRI [0–100]
→ Reliability caveat
SRI < 40 → LOW_CAPTURE
_RELIABILITY_
RECOMMEND_RECAPTURE
Clinical Safety Guarantee: Because urgency is derived exclusively from PRI, a high-confidence pathology finding triggers IMMEDIATE_CLINICAL_PRIORITY even when the user is maximally fatigued (eye_fatigue = 100). Fatigue affects only the SRI output and its advisory caveat — it can never downgrade a genuine pathology alert. This is enforced structurally, not by a rule patch.
Verified by automated test: test_high_pathology_triggers_urgency_regardless_of_fatigue parametrises eye_fatigue across [0, 50, 98, 100] and asserts IMMEDIATE_CLINICAL_PRIORITY in all cases when cnn_confidence = 0.92.
Membership Functions
Triangular MFs for mid-range values (Low, Medium, High). Trapezoidal (edge-pinned shoulder) MFs at universe boundaries to guarantee full coverage — no gap where inputs could fall outside all MFs and produce NaN.
Rule Structure
Expert-defined 3×3 rule blocks (9 rules per system). PRI rules: if confidence is High AND symptoms are Moderate → PRI is High. All rule antecedents use fuzzy AND (minimum t-norm).
Defuzzification
Centroid method (centre of gravity). If the aggregated output is non-finite (edge case NaN from skfuzzy), a NaN guard catches it and returns a neutral fallback value of 50, preventing any unhandled exception propagating to the API.

GDPR Compliance: All image processing occurs in-memory on the local server. No image bytes are written to disk or transmitted to any external service. The database stores only derived numerical scores. Users can delete their session history at any time from the History page.
